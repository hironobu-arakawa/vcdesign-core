# 2. Operation の再定義
長い間、
運用（Operation）は「作った後に残る作業」と考えられてきました。
- 障害対応
- 監視
- 保守
- 小さな改修
開発が主役で、
運用はその後始末のように扱われることも少なくありません。
しかし、価値の断絶が
完成でも障害でもなく、
運用という時間の中で起きていると気づいたとき、
この見方は成り立たなくなりました。

## 運用は「維持」ではなかった
運用の現場で行われているのは、
単なる状態維持ではありません。
現実は、常に変わります。
- 利用のされ方が変わる
- 周囲の業務が変わる
- 社会や規制が変わる
- 技術の前提が変わる
それでもシステムは動き続ける。
そのたびに、
どこまでをシステムが引き受け、
どこからを人に返すか
という判断が、静かに行われています。
この判断こそが、
価値を続けるか、断絶させるかを分けていました。

## 運用で起きているのは「判断の更新」
運用で起きていることを
「変更」や「改善」と呼ぶのは簡単です。
しかし本質は、
過去の判断を、いまの現実に照らして引き受け直すことでした。
- かつては妥当だった自動化
- かつては合理的だったルール
- かつては安全だった前提
それらが揺らいだとき、
誰かが判断します。
- まだ引き受けられるか
- どこまでなら引き受けられるか
- もう引き受けるべきではないか
この判断が行われる場所が、運用です。

## 「作る」ことが一瞬で終わる世界で
AI によって、
実装のコストは限りなくゼロに近づきました。
正しそうな設計は、
いつでも作れます。
最適そうな構造も、
何度でも試せます。
だからこそ、
価値の重心は静かに移動しました。
作ることから、
引き受け続けることへ。
運用とは、
完成品を守る行為ではありません。
完成してしまった判断を、
これ以上引き受け続けてよいかを問い直す行為です。

## Operation を再定義する
VCDesign における Operation とは、
- システムを安定稼働させることでも
- 障害を減らすことでもありません。
Operation とは、
章 をまたいで、
判断と責任を切らさずに手渡し続ける行為です。
引き受ける判断がある。
返す判断がある。
その境界を見極める。
その積み重ねだけが、
価値を時間の中で生かし続けます。

## 運用は、最も創造的な仕事になる
作ることが容易になった世界では、
最も難しいのは、
「作らない判断」をすることです。
- どこまで自動化しないか
- どこで人に返すか
- どの責任を、次の 章 に渡すか
これらを誤れば、
システムは壊れなくても、
価値は壊れます。
だから、VCDesign では言います。
Operation は、
最も遅れてやってくる創造行為である。

## 次へ
ここまでで、
私たちは気づきました。
- 章 が、価値を渡す単位であるなら
- Operation は、それを渡し続ける行為である
では、その行為を引き受けるのは、誰なのか。
次章では、
「Builder」でも「運用担当」でもない、
新しいエンジニアの役割について扱います。
