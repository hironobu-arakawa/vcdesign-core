# docs/

このディレクトリは、VCDesign を**読むための空間**です。  
仕様（specs）やルール（policies）を説明する場所ではありません。

ここにある文書はすべて、
- 何を実装するか
- どう最適化するか

ではなく、

> **どの位置から、VCDesign を読むのか**

を揃えるために置かれています。

---

## 構成と役割

### prelude/
VCDesign を読む前に立つ場所です。

- これから扱う抽象度
- 前提となる態度
- 読み進める際の注意点

を静かに揃えます。

**最初に読む場所**です。

---

### philosophy/
なぜ VCDesign が必要になったのか。  
どのような問題意識から生まれたのか。

思想的な背景を扱いますが、
答えや正解を与えることは目的ではありません。

---

### narrative/
VCDesign の考え方が、
どのような文脈で立ち上がり、
どのように理解されていくかを辿ります。

設計論ではなく、
**理解のプロセス**を扱います。

---

### manifest/
VCDesign が何を宣言し、
何を宣言しないのか。

立場表明を明確にするための文書群です。

---

### analogy/
これは少し異色のディレクトリです。

prelude を読んでもなお抽象すぎて掴めない人のために、
日常的な比喩を使って**感覚的なジャンプ**を行うための文書を置いています。

- 概念を説明する場所ではありません
- 理解を強制する場所でもありません

それでも掴めない場合の、
**最後の橋渡し**として存在します。

---

## 読む順序について

原則として、以下の順で読むことを想定しています。

1. prelude/
2. philosophy/
3. narrative/
4. manifest/

もし prelude の時点で抽象度が高すぎると感じた場合は、  
**analogy/ から読んでも構いません。**

---

## 注意

この docs 以下の文書は、
- 仕様の正本ではありません
- 判断ルールでもありません

VCDesign の正本仕様は `specs/vcdesign_core.yaml` にあります。

ここはあくまで、
**読む人が立つ位置を揃えるための場所**です。
